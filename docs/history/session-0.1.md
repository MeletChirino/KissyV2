# Session 0.1 — Toolchain bootstrap

**Date:** 2026-07-08
**Status:** completed
**Scope:** Toolchain only. No behavior changes to `src/telegram_bot/*`.

## Goals

- Replace ad-hoc tooling with a single `uv`-based workflow.
- Lock Python version to 3.12.
- Add a minimal test harness that proves the toolchain works end-to-end.
- Add `docs/history/` so every future session has a written record.
- Update top-level docs so a fresh contributor (or future-you) can onboard in 5 minutes.

## Decisions made this session

| Topic | Decision | Why |
|---|---|---|
| Toolchain manager | `uv` | Single binary replaces `pip` + `venv` + lockfile. Beginner-friendly: `uv sync` does it all. |
| Python version | 3.12 | Stable, supported until 2028, broad library compatibility. |
| Test runner | `pytest` + `pytest-asyncio` | De facto standard; needed for `python-telegram-bot` async handlers later. |
| HTTP mocking for tests | `respx` | Mock the Invoice Ninja + Ollama clients cleanly. |
| Linter / formatter | `ruff check` + `ruff format` | One Rust tool replaces `flake8` + `isort` + `black` + plugins. |
| Typechecker | `mypy --strict` | Mature, first-class `pydantic` support for the LLM JSON schemas. |
| Build metadata | `pyproject.toml` (PEP 621) | Modern standard. Single source of truth for ruff, mypy, pytest, and deps. |
| `requirements.txt` | **Removed** | Replaced by `pyproject.toml` + `uv.lock`. |
| Branches | `master` (per owner) | Solo project, no PR review needed. Owner commits after each session. |
| `uv` install on prod | OS-agnostic script with fallbacks | macOS uses `curl \| sh`; Debian uses `apt` repo or fallback. |

## Commands the owner will run (in order, on the Debian server)

```bash
# 1. Install uv (idempotent; safe to re-run).
bash scripts/install_uv.sh

# 2. From the repo root: install runtime + dev deps and create .venv.
uv sync

# 3. Prove the toolchain works.
uv run pytest            # 1 passed
uv run ruff check .      # all clean (or only known TODO items)
uv run ruff format --check .
uv run mypy src tests    # may surface pre-existing type issues; logged below

# 4. Daily driver: run the bot.
uv run python -m telegram_bot
```

## Addendum (same session, owner feedback) — Ninja image missing from the plan

**Caught by the owner:** Session 0.1 originally scoped the toolchain only
and never mentioned the Invoice Ninja Docker stack. That was a real gap;
the MVP is a two-piece system (bot + Ninja) and both need to be running
for the bot to do anything useful.

**Resolution (in this same session, before commit):**

- Confirmed the Ninja stack is already defined in `docker-compose.yaml`
  (services: `app`, `nginx`, `mysql`, `redis`). Nothing was missing from
  the image side.
- What was missing was the **operational runbook**: how the bot reaches
  Ninja, in what order to start things, and what the URL topology looks
  like. The fix is below.

**Files added in this addendum:**

- `scripts/start_ninja.sh` — idempotent runbook that pulls the images,
  brings the stack up, waits for `app` to be healthy, and reminds the
  owner to fetch a static API token from the Ninja UI.
- `docs/ninja-bot-integration.md` — explains the topology: bot on host,
  Ninja in Docker, loopback HTTP between them, and why we are NOT putting
  the bot in Docker for MVP.
- `scripts/systemd/kissy-bot.service.example` — starter unit for running
  the bot as a service on the Debian box. Unprivileged user, hardened
  filesystem, `uv run` so no manual venv activation.
- `README.md` "Run the full MVP stack" section now lists the two-piece
  startup order (Ninja first, bot second) and links to the integration
  doc.

**Architecture decision recorded:** the bot stays on the host for MVP.
Moving it into Docker (a fifth service in `docker-compose.yaml`) is
deferred to post-MVP per `AGENTS.md`.
